## Overview

The FeatureFlags module provides a simple yet powerful feature flag system with ChatOps integration for managing features at runtime without code changes.

## Installation

```bash
npm install big-world-bigger-ideas
```

## Basic Usage

```javascript
const { FeatureFlags } = require('big-world-bigger-ideas');

// Initialize with config file path
const flags = new FeatureFlags('./feature-flags.json');

// Check if a feature is enabled
if (flags.isEnabled('new-dashboard')) {
  // Show new dashboard
} else {
  // Show old dashboard
}
```

## Constructor

### `new FeatureFlags(configPath?)`

Creates a new FeatureFlags instance.

**Parameters:**

<ParamField path="configPath" type="string" optional default="./feature-flags.json">
  Path to the feature flags JSON configuration file
</ParamField>

## Methods

### `isEnabled(flagName)`

Check if a feature flag is enabled.

**Parameters:**

<ParamField path="flagName" type="string" required>
  The name of the feature flag to check
</ParamField>

**Returns:** `boolean` - True if enabled, false otherwise

**Example:**

```javascript
if (flags.isEnabled('beta-features')) {
  console.log('Beta features are enabled!');
}
```

### `setFlag(flagName, enabled)`

Enable or disable a feature flag.

**Parameters:**

<ParamField path="flagName" type="string" required>
  The name of the feature flag
</ParamField>

<ParamField path="enabled" type="boolean" required>
  Whether to enable or disable the flag
</ParamField>

**Example:**

```javascript
// Enable a feature
flags.setFlag('new-feature', true);

// Disable a feature
flags.setFlag('old-feature', false);
```

### `listFlags()`

Get all feature flags and their status.

**Returns:** `Object` - All flags with their enabled status and timestamps

**Example:**

```javascript
const allFlags = flags.listFlags();
console.log(JSON.stringify(allFlags, null, 2));
```

## Configuration File

The feature flags are stored in a JSON file:

```json feature-flags.json
{
  "flags": {
    "new-dashboard": {
      "enabled": true,
      "updatedAt": "2024-01-15T10:30:00.000Z"
    },
    "beta-features": {
      "enabled": false,
      "updatedAt": "2024-01-10T08:00:00.000Z"
    }
  }
}
```

## ChatOps Integration

Manage feature flags via GitHub comments using ChatOps commands:

### Enable a Feature

Comment on any issue or PR:

```
/chatops run feature set new-dashboard
```

### Disable a Feature

```
/chatops run feature unset new-dashboard
```

### List All Features

```
/chatops run feature list
```

## Usage Patterns

### Gradual Rollout

```javascript
const flags = new FeatureFlags();

// Start with feature disabled
flags.setFlag('new-algorithm', false);

// Test in development
if (process.env.NODE_ENV === 'development') {
  flags.setFlag('new-algorithm', true);
}

// Gradually roll out to production via ChatOps
```

### A/B Testing

```javascript
const flags = new FeatureFlags();

function getFeatureVariant(userId) {
  if (!flags.isEnabled('ab-test-enabled')) {
    return 'control';
  }
  
  // Split users into test groups
  return userId % 2 === 0 ? 'variant-a' : 'variant-b';
}
```

### Kill Switch

```javascript
const flags = new FeatureFlags();

async function processData() {
  // Check kill switch before expensive operation
  if (!flags.isEnabled('data-processing')) {
    console.log('Data processing disabled');
    return;
  }
  
  // Proceed with processing
  await heavyDataProcessing();
}
```

## Best Practices

<AccordionGroup>
  <Accordion title="Use descriptive names">
    Use clear, descriptive names for your feature flags:
    - ✅ `new-checkout-flow`
    - ✅ `enable-caching`
    - ❌ `feature1`
    - ❌ `test`
  </Accordion>

  <Accordion title="Clean up old flags">
    Remove feature flags once they're fully rolled out or deprecated to avoid clutter.
  </Accordion>

  <Accordion title="Default to safe values">
    When a flag doesn't exist, default to the safe/stable behavior.
  </Accordion>

  <Accordion title="Document your flags">
    Keep a README or wiki documenting what each flag does and when it was added.
  </Accordion>
</AccordionGroup>
